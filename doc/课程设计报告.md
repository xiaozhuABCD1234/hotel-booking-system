## 一、需求分析

### 1.1 问题描述

本系统为上海电力大学数据库原理课程设计选题四——酒店预订管理系统。系统模拟一个第三方酒店预订平台（类似携程），支持用户注册、酒店搜索、客房预订、订单管理、评价及数据统计等功能。

系统采用前后端分离架构，技术栈如下：

| 层级 | 技术选型 |
|------|----------|
| 后端 | Go + Fiber v3 + GORM |
| 前端 | Vue 3 + TypeScript + Vite |
| UI 框架 | Tailwind CSS v4 + Reka UI |
| 数据库 | PostgreSQL 18 |
| 认证 | JWT（golang-jwt） |
| API 文档 | Swaggo / Swagger |

系统目标是实现一个功能完整、数据安全、具备积分与 VIP 优惠体系的酒店预订管理平台，涵盖从用户注册、客房搜索、下单预订、入住管理、评价反馈到后台数据统计的完整业务流程。

### 1.2 系统功能描述

#### 功能需求

1. **用户管理** — 用户注册、登录、个人信息维护；系统内置积分体系，用户完成订单后自动累积积分，积分达到阈值自动升级 VIP 等级，享受对应折扣优惠。
2. **客房预订** — 支持为自己或他人预订客房，一次可预订多间，下单时需确认入住人身份证、姓名等信息，系统自动校验库存并扣减可预订数量。
3. **酒店搜索** — 按城市、区域、酒店名称、价格范围、入住日期等多条件筛选酒店，支持模糊搜索与分页查询。
4. **房型选择** — 进入酒店详情页后展示多种房型（含价格、库存、设施、图片），供用户按需选择。
5. **订单管理** — 用户可查询全部历史订单，支持按状态（已预订、已入住、已完成、已取消等）分类筛选；订单状态机为 `pending → booked → checked_in → completed`，可从 `booked` 或 `checked_in` 状态取消。
6. **用户评价** — 用户可对已完成的入住订单进行评分（1～5星）和文字评价，系统校验评价用户与订单预订人一致性。
7. **退订** — 已预订或已入住但未完成的订单可申请退订，退订后客房库存自动恢复，已完成订单的积分相应扣减。
8. **后台维护** — 管理员可维护系统后台信息，包括酒店信息、客房房型、库存数量、价格等。
9. **汇总统计** —
    - 按城市、区域、酒店、价格等因素统计客房预订情况与入住率；
    - 按入住人信息（年龄、性别、职业、学历、收入等）统计预订情况，分析客户偏好。

#### 数据实体

系统需存储以下核心数据实体：

| 数据实体 | 说明 |
|----------|------|
| 用户（user） | 账户持有者，含用户名、密码哈希、手机号、邮箱、真实姓名、身份证号、角色、积分、VIP等级、职业、学历、收入 |
| VIP等级（vip_level） | 等级定义表，含等级编号、等级名称、最低积分、折扣率 |
| 地区（region） | 省/市/区三级层次结构，自引用外键 |
| 酒店（hotel） | 酒店基本信息，含名称、所属地区、地址、电话、星级、描述 |
| 酒店图片（hotel_image） | 酒店图片 URL，与酒店多对一 |
| 客房/房型（room） | 房型定义，含所属酒店、房型名称、总数量、可预订数量、平日价、周末价 |
| 客房图片（room_image） | 客房图片 URL，与客房多对一 |
| 客房设施（room_facility） | 客房设施列表，与客房多对一 |
| 人员（person） | 入住人身份信息，以身份证号为主键，含姓名、电话 |
| 订单（order） | 预订订单，含下单用户、房型、数量、入住/离店日期、总价、实付价、状态 |
| 入住人关联（order_guest） | 订单与入住人的多对多关联表 |
| 评价（review） | 用户对订单的评价，含评分、内容，约束同一用户对同一订单只能评价一次 |
| JWT黑名单（jwt_blacklist） | 已注销的 JWT 令牌标识，用于登出功能 |

### 1.3 数据库完整性要求

#### 1.3.1 实体完整性

系统采用三种主键策略，确保每张表的实体完整性：

| 主键类型 | 适用表 | 说明 |
|----------|--------|------|
| UUID v7 | `user_1718`、`hotel_1718`、`room_1718`、`order_1718`、`review_1718` | 使用 `uuidv7()` 函数生成，兼顾全局唯一性与时间有序性 |
| CHAR(18) | `person_1718` | 以18位身份证号作为自然主键 |
| SERIAL | `region_1718` | 自增整数主键，适用于地区这类静态维度表 |
| 复合主键 | `hotel_image_1718`、`room_image_1718`、`room_facility_1718`、`order_guest_1718` | 由两个外键字段组成联合主键 |

此外，`jwt_blacklist_1718` 表以 `jti TEXT`（JWT 令牌唯一标识）作为主键。

#### 1.3.2 参照完整性

系统定义了以下外键约束，确保表间关联数据的一致性：

| 子表 | 外键字段 | 引用父表 | 删除策略 |
|------|----------|----------|----------|
| `user_1718` | `vip_level` | `vip_level_1718(level)` | 默认（RESTRICT） |
| `hotel_1718` | `region_id` | `region_1718(id)` | 默认（RESTRICT） |
| `hotel_image_1718` | `hotel_id` | `hotel_1718(id)` | ON DELETE CASCADE |
| `room_1718` | `hotel_id` | `hotel_1718(id)` | ON DELETE CASCADE |
| `room_image_1718` | `room_id` | `room_1718(id)` | ON DELETE CASCADE |
| `room_facility_1718` | `room_id` | `room_1718(id)` | ON DELETE CASCADE |
| `order_1718` | `user_id` | `user_1718(id)` | 默认（RESTRICT） |
| `order_1718` | `room_id` | `room_1718(id)` | 默认（RESTRICT） |
| `order_guest_1718` | `order_id` | `order_1718(id)` | ON DELETE CASCADE |
| `order_guest_1718` | `id_card` | `person_1718(id_card)` | 默认（RESTRICT） |
| `review_1718` | `user_id` | `user_1718(id)` | 默认（RESTRICT） |
| `review_1718` | `order_id` | `order_1718(id)` | 默认（RESTRICT） |
| `review_1718` | `hotel_id` | `hotel_1718(id)` | 默认（RESTRICT） |
| `region_1718` | `parents_id` | `region_1718(id)` | 默认（RESTRICT，自引用） |

级联删除设计说明：酒店删除时，其下的图片、客房及客房附属数据（图片、设施）一并删除；订单删除时，其入住人关联记录一并删除。用户、房型等核心实体采用 RESTRICT 策略，防止误删产生数据孤岛。

#### 1.3.3 用户定义完整性

系统通过 CHECK 约束实现丰富的业务规则校验：

**身份证号校验**
- `user_1718.id_card`：`CHECK (id_card ~ '^\d{17}[\dXx]$' OR id_card IS NULL)`
- `person_1718.id_card`：`CHECK (id_card ~ '^\d{17}[\dXx]$')`（主键，不允许为空）

**电话号码校验**
- `person_1718.phone`：`CHECK (phone ~ '^\+?[0-9\-]+$')`

**星级约束**
- `hotel_1718.star_level`：`CHECK (star_level BETWEEN 1 AND 5)`

**评分约束**
- `review_1718.rating`：`CHECK (rating BETWEEN 1 AND 5)`

**日期约束**
- `order_1718`：`CHECK (check_out_date > check_in_date)`，确保离店日期晚于入住日期

**库存约束**
- `room_1718.total_quantity`：`CHECK (total_quantity > 0)`
- `room_1718.available_quantity`：`CHECK (available_quantity >= 0)`
- `room_1718`：`CHECK (available_quantity <= total_quantity)`，确保可预订数量不超过总量

**价格约束**
- `room_1718.price`：`CHECK (price > 0)`
- `room_1718.weekend_price`：`CHECK (weekend_price > 0)`
- `order_1718.total_price`：`CHECK (total_price >= 0)`
- `order_1718.actual_price`：`CHECK (actual_price >= 0)`

**数量约束**
- `order_1718.quantity`：`CHECK (quantity > 0)`
- `vip_level_1718.level`：`CHECK (level >= 0)`
- `vip_level_1718.min_points`：`CHECK (min_points >= 0)`
- `vip_level_1718.discount_rate`：`CHECK (discount_rate BETWEEN 0 AND 1)`

**积分约束**
- `user_1718.points`：`CHECK (points >= 0)`

**唯一约束**
- `review_1718`：`UNIQUE (user_id, order_id)`，同一用户对同一订单只能评价一次

**默认值约束**
- `user_1718.role` 默认 `'customer'`
- `user_1718.points` 默认 `0`
- `user_1718.vip_level` 默认 `0`
- `user_1718.status`、`hotel_1718.status`、`room_1718.status` 默认 `1`（活跃状态）
- `order_1718.status` 默认 `'pending'`
- 各表 `create_at`、`update_at` 默认 `now()`

#### 1.3.4 触发器完整性保障

系统共定义 9 个触发器，分为业务触发器和维护触发器两类。

**业务触发器（4个）**

1. **`trg_review_user_match_1718`** — 评价用户一致性校验
    - 触发时机：`BEFORE INSERT OR UPDATE OF user_id, order_id ON review_1718`
    - 调用函数：`fn_validate_review_user_1718()`
    - 功能：校验评价的 `user_id` 必须与关联订单的 `user_id` 一致，防止用户为他人的订单撰写评价。

2. **`trg_order_room_quantity_1718`** — 客房库存自动管理
    - 触发时机：`AFTER INSERT OR UPDATE OF status ON order_1718`
    - 调用函数：`fn_update_room_quantity_1718()`
    - 功能：订单状态变为 `booked` 或 `checked_in` 时，自动扣减对应房型的可预订数量（含库存不足校验）；订单取消时，自动恢复库存。

3. **`trg_order_user_points_1718`** — 用户积分与VIP等级自动更新
    - 触发时机：`AFTER INSERT OR UPDATE OF status ON order_1718`
    - 调用函数：`fn_update_user_points_1718()`
    - 功能：订单状态变为 `completed` 时，自动将订单实付金额取整后累加到用户积分；订单从 `completed` 变为其他状态时，扣减对应积分。每次积分变更后，自动根据用户当前积分查询 `vip_level_1718` 表，更新用户的 VIP 等级。

4. **`trg_order_vip_discount_1718`** — 下单自动应用VIP折扣
    - 触发时机：`BEFORE INSERT ON order_1718`
    - 调用函数：`fn_apply_vip_discount_1718()`
    - 功能：新订单插入前，根据下单用户的 VIP 等级查询对应折扣率，自动计算 `actual_price = total_price × discount_rate`，保证折扣应用的原子性，避免并发竞态问题。

**维护触发器（5个）**

以下 5 个触发器均调用同一函数 `fn_set_update_at_1718()`，在对应表执行 `UPDATE` 操作前自动将 `update_at` 字段刷新为当前时间戳：

5. **`trg_user_update_at_1718`** — `BEFORE UPDATE ON user_1718`
6. **`trg_hotel_update_at_1718`** — `BEFORE UPDATE ON hotel_1718`
7. **`trg_room_update_at_1718`** — `BEFORE UPDATE ON room_1718`
8. **`trg_order_update_at_1718`** — `BEFORE UPDATE ON order_1718`
9. **`trg_review_update_at_1718`** — `BEFORE UPDATE ON review_1718`

#### 1.3.5 存储过程与函数

系统定义了 4 个存储过程/函数：

1. **`fn_query_orders_by_user_1718(p_username TEXT, p_id_card CHAR(18))`** — 按用户信息查询订单
    - 功能：支持按用户名模糊匹配或身份证号精确匹配查询关联订单。先通过身份证号在 `order_guest_1718` 中查找关联的预订用户ID，再合并用户名匹配的用户ID，最终返回这些用户的所有订单详情（含酒店名、房型、数量等）。
    - 返回：订单ID、入住/离店日期、状态、总价、实付价、创建时间、酒店名、房型名、数量。

2. **`fn_room_occupancy_rate_1718(p_hotel_id UUID, p_start_date DATE, p_end_date DATE)`** — 客房入住率统计
    - 功能：统计指定酒店在指定时间段内各房型的入住率。入住率计算公式为：已预订房晚数 ÷（总客房数 × 统计天数）× 100。同时计算各房型的营收额。
    - 返回：房型ID、房型名、总数量、可预订数量、已预订房晚数、入住率百分比、单价、总营收。

3. **`sp_ensure_person_1718(p_id_card CHAR(18), p_name TEXT, p_phone VARCHAR(20))`** — 确保入住人存在
    - 功能：存储过程，在创建订单前调用。若入住人身份证号已存在于 `person_1718` 表中，则更新其电话号码；否则插入新记录。使用 `ON CONFLICT ... DO UPDATE` 实现幂等操作，避免 `order_guest_1718` 外键约束失败。

4. **`fn_order_detail_1718(p_order_id UUID)`** — 订单详情查询
    - 功能：查询指定订单的完整详情，关联用户信息、酒店信息、房型信息及入住人列表。入住人姓名通过 `STRING_AGG` 聚合为逗号分隔的字符串。
    - 返回：订单ID、状态、数量、入住/离店日期、天数、总价、实付价、创建时间、预订用户名/真实姓名/电话、酒店名、房型名、房价、入住人数、入住人姓名列表。

### 1.4 社会、安全、法律等因素及数据库安全性设置

#### 1.4.1 社会因素

酒店预订管理系统涉及大量用户个人信息（身份证号、手机号、真实姓名、职业、学历、收入等），属于敏感个人数据。系统在设计与实现中需充分考虑以下社会因素：

- **个人信息保护**：系统存储用户的身份证号、手机号、真实姓名等敏感信息，必须严格限制数据访问范围，仅授权人员可查看，防止信息泄露导致用户遭受骚扰或诈骗。
- **数据最小化原则**：系统仅收集业务必需的个人信息，不强制要求用户填写所有字段（如身份证号允许为空），尊重用户的数据自主权。
- **数据隐私合规**：入住人信息与订单预订人信息分离存储（`person_1718` 与 `user_1718`），预订人可为他人预订，但入住人的个人信息独立管理，减少不必要的数据关联。
- **透明度**：用户可随时查看和修改自己的个人信息，了解积分与VIP等级的计算规则。

#### 1.4.2 安全因素

系统在多个层面实施安全防护：

- **JWT 认证机制**：系统采用 JSON Web Token（JWT）实现无状态身份认证。认证流程如下：
    - 用户登录后，服务端签发 Access Token（有效期15分钟）和 Refresh Token（有效期7天）；
    - 令牌使用 HS256 算法（HMAC-SHA256）签名，签名密钥要求至少16字节，通过环境变量 `JWT_SECRET` 配置；
    - 每个令牌包含唯一的 `jti`（JWT ID），支持令牌级别的细粒度管理；
    - 请求时通过 `Authorization: Bearer <token>` 头携带令牌，中间件验证签名、有效期及黑名单状态。
- **密码安全存储**：用户密码使用 BCrypt 算法进行单向哈希存储（存储字段为 `VARCHAR(255)`），数据库中不保存明文密码，即使数据库泄露也无法直接获取用户密码。
- **访问控制**：系统基于用户角色（`user_role` 枚举：`customer`、`hotel_manager`、`admin`）实施权限控制，不同角色可访问的接口和数据范围不同。
- **令牌注销与黑名单**：用户登出时，当前令牌的 `jti` 被写入 `jwt_blacklist_1718` 表；认证中间件在每次请求时检查令牌是否在黑名单中，已注销的令牌将被拒绝，防止令牌被盗用后的持续访问风险。
- **SQL 注入防护**：系统使用 GORM 作为 ORM 框架，所有数据库操作通过参数化查询执行，从根源上杜绝 SQL 注入攻击。

#### 1.4.3 法律因素

系统涉及个人信息处理，需遵守相关法律法规：

- **数据保护法规**：系统处理的身份证号、手机号等属于《中华人民共和国个人信息保护法》定义的敏感个人信息，需获得用户明确同意方可收集和处理，并采取严格的安全保护措施。
- **用户知情权**：用户有权了解其个人信息的收集范围、使用目的和存储方式；用户有权查询、更正、删除其个人信息。
- **数据存储与删除**：系统采用软删除机制（`status` 字段：1=活跃，0=已删除），用户数据不会被物理删除，便于数据恢复和审计追溯。若用户依法请求删除其个人信息，应确保相关数据被彻底清除或匿名化处理。
- **数据访问审计**：系统记录关键操作的时间戳（`create_at`、`update_at`），为数据访问和变更提供可追溯的审计线索。

#### 1.4.4 数据库安全性设置

系统在数据库层面实施了以下安全措施：

- **认证机制**：所有数据库操作通过后端应用层发起，应用通过 `.env` 配置文件中的数据库连接凭证（用户名、密码、主机、端口、数据库名）连接 PostgreSQL，凭证不硬编码在源码中。
- **授权机制**：系统基于角色的访问控制（RBAC）通过 `user_1718.role` 字段（`customer`、`hotel_manager`、`admin`）实现，后端中间件在处理请求前校验用户角色，确保只有授权角色可执行特定操作（如后台维护功能仅限 `admin` 和 `hotel_manager`）。
- **密码存储安全**：用户密码使用 BCrypt 算法哈希后存储于 `user_1718.password` 字段（`VARCHAR(255)`），数据库中不存在明文密码。
- **JWT 黑名单登出**：`jwt_blacklist_1718` 表存储已注销令牌的 `jti` 和过期时间，认证中间件 `JWTAuth` 在每次请求时查询该表，拒绝已被注销的令牌。表上建有 `idx_blacklist_expires_1718` 索引（按 `expires_at`），便于定期清理过期记录。
- **软删除策略**：核心业务表（`user_1718`、`hotel_1718`、`room_1718`）使用 `status` 字段（1=活跃，0=已删除）实现逻辑删除，避免数据被物理移除，保留审计追溯能力。
- **触发器数据校验**：`trg_review_user_match_1718` 触发器在数据库层面强制校验评价用户与订单预订人的一致性，即使应用层校验被绕过，数据库层也能拦截非法数据。

## 二、概念结构设计

### 2.1 系统整体E-R模型

本系统采用实体-联系（E-R）模型进行概念结构设计。系统共包含 **13 个实体**，分别为：VIP等级（VipLevel）、用户（User）、地区（Region）、酒店（Hotel）、酒店图片（HotelImage）、客房/房型（Room）、客房图片（RoomImage）、客房设施（RoomFacility）、人员/入住人（Person）、订单（Order）、订单入住人关联（OrderGuest）、评价（Review）、JWT黑名单（JwtBlacklist）。

#### 2.1.1 实体间联系

系统中各实体之间的联系如下表所示：

| 编号 | 联系 | 基数 | 说明 |
|:---:|------|:---:|------|
| 1 | User → Order | 1:N | 一个用户可以下多个订单，一个订单只属于一个用户 |
| 2 | Order → Room | N:1 | 一个订单对应一个房型，一个房型可被多个订单预订 |
| 3 | Room → Hotel | N:1 | 一个房型属于一个酒店，一个酒店拥有多个房型 |
| 4 | Hotel → Region | N:1 | 一个酒店位于一个地区，一个地区下有多个酒店 |
| 5 | Region → Region（自引用） | N:1 | 地区表通过 `parents_id` 自引用，形成省→市→区三级层次结构；一个地区属于一个上级地区，一个上级地区下辖多个下级地区 |
| 6 | Order ↔ Person（经 OrderGuest） | N:M | 一个订单可有多个入住人，一个入住人可出现在多个订单中；通过关联表 `order_guest_1718` 实现多对多联系 |
| 7 | User → Review | 1:N | 一个用户可以写多条评价，一条评价只属于一个用户 |
| 8 | Review → Hotel | N:1 | 一条评价针对一个酒店，一个酒店可有多条评价 |
| 9 | Review → Order | N:1 | 一条评价关联一个订单，一个订单最多对应一条评价（同一用户对同一订单只能评价一次，受 UNIQUE 约束） |
| 10 | User → VipLevel | N:1 | 多个用户可属于同一VIP等级，一个用户只有一个VIP等级 |
| 11 | Hotel → HotelImage | 1:N | 一个酒店可有多张图片，一张图片只属于一个酒店 |
| 12 | Room → RoomImage | 1:N | 一个房型可有多张图片，一张图片只属于一个房型 |
| 13 | Room → RoomFacility | 1:N | 一个房型可有多个设施，一条设施记录只属于一个房型 |

#### 2.1.2 设计要点说明

- **用户与人员分离**：`user_1718`（用户）是系统账户持有者，负责下单、评价等操作；`person_1718`（人员）是实际入住人，记录身份证号、姓名、手机号等身份信息。两者通过 `order_guest_1718` 关联表建立多对多联系，支持用户为他人预订。
- **评价唯一性约束**：`review_1718` 表上设有 `UNIQUE(user_id, order_id)` 约束，确保同一用户对同一订单只能评价一次。
- **地区层次结构**：`region_1718` 表通过 `parents_id` 外键自引用，形成省→市→区三级树形结构，支持按不同行政层级进行酒店筛选和统计。
- **级联删除策略**：酒店图片、客房图片、客房设施、订单入住人关联等附属表在外键上设置了 `ON DELETE CASCADE`，当主实体被删除时自动清理关联记录；而用户、客房、酒店等核心业务表使用默认的 `RESTRICT` 策略，防止误删有业务数据依赖的记录。
- **JWT黑名单**：`jwt_blacklist_1718` 用于存储已注销的JWT令牌标识（`jti`），实现用户登出后令牌即时失效，不属于核心业务实体，但在系统安全架构中不可或缺。

### 4.2 索引作用分析

#### 用户表索引

- **`idx_user_phone_1718`** — 手机号是系统登录的主要凭证，用户登录时后端通过 `WHERE phone = ?` 精确查找用户记录，该索引将登录查询从全表扫描优化为索引点查。
- **`idx_user_role_1718`** — 管理员后台按角色筛选用户（如列出所有酒店管理员），`WHERE role = 'hotel_manager'` 的过滤条件通过该索引加速。
- **`idx_user_points_1718` (DESC)** — 后台按积分降序排列用户以展示 VIP 排行榜，降序索引避免了排序操作，直接按索引顺序扫描即可获得积分从高到低的结果。
- **`idx_user_income_1718`** — 收入字段用于客户偏好统计分析（如不同收入层次的预订偏好），该索引加速 `WHERE income BETWEEN ... AND ...` 的范围查询。

#### 地区表索引

- **`idx_region_parents_1718`** — 地区表通过 `parents_id` 自引用实现省→市→区三级层次结构。查询某市下所有区时 `SELECT ... WHERE parents_id = ?`，该索引将递归/层次查询优化为索引范围扫描。

#### 酒店表索引

- **`idx_hotel_region_1718`** — 酒店搜索最常见的场景是按城市/区域筛选，`WHERE region_id IN (子查询获取某省/市的所有子地区ID)` 通过该索引加速，是酒店搜索的首个过滤条件。
- **`idx_hotel_name_1718`** — 支持按酒店名称模糊搜索（`WHERE hotel_name LIKE '%关键词%'`），为 B-Tree 索引虽然不能直接加速模糊搜索，但在前缀匹配场景（`酒店名%`）中有显著效果。
- **`idx_hotel_star_1718`** — 用户可按星级筛选酒店（如只看四星级以上），该索引加速 `WHERE star_level >= 4` 的范围查询。

#### 客房表索引

- **`idx_room_hotel_1718`** — 查询某酒店下所有房型时执行 `WHERE hotel_id = ?`，是酒店详情页的核心查询路径。
- **`idx_room_type_1718`** — 支持按房型名称搜索（如"豪华套房"），加速 `WHERE type_name = ?` 精确匹配查询。
- **`idx_room_price_1718`** — 加速价格范围过滤（`WHERE price BETWEEN ? AND ?`）和按价格排序展示（`ORDER BY price`），是酒店搜索结果排序的核心支撑。
- **`idx_room_available_1718`** — 过滤有库存的房型（`WHERE available_quantity > 0`），避免展示已售罄的房型给用户。

#### 订单表索引

- **`idx_order_user_1718`** — "我的订单"页面的核心索引，`WHERE user_id = ? ORDER BY create_at DESC` 查询通过该索引大幅加速。
- **`idx_order_room_1718`** — 房型销量统计查询使用 `WHERE room_id = ?` 关联计算各房型的入住率，用于后台统计报表（`fn_room_occupancy_rate_1718` 函数依赖此索引）。
- **`idx_order_status_1718`** — 用户和管理员按状态筛选订单（如只看"已完成"订单），`WHERE status = 'completed'` 通过该索引加速。
- **`idx_order_dates_1718`** (复合索引) — 加速按日期范围查询订单（`WHERE check_in_date BETWEEN ? AND ?`），是入住率统计和日期维度数据分析的核心索引；同时复合索引中第一个键 `check_in_date` 也可单独用于按入住日期排序。
- **`idx_order_create_1718`** — 订单列表默认按创建时间降序排列，该索引加速 `ORDER BY create_at DESC`，避免显式排序。

#### 评价表索引

- **`idx_review_hotel_1718`** — 酒店详情页展示评价列表时使用 `WHERE hotel_id = ?`，是该页面加载的核心索引。
- **`idx_review_user_1718`** — 用户个人中心展示历史评价使用 `WHERE user_id = ?`，加速用户的评价记录查询。
- **`idx_review_rating_1718`** — 支持按评分过滤（如只看4分以上评价），`WHERE rating >= 4` 的范围查询通过该索引优化。

#### 入住人关联表索引

- **`idx_order_guest_id_card_1718`** — 按身份证号查询入住人的所有历史订单记录，用于客服查询和后台客户分析。

#### JWT黑名单表索引

- **`idx_blacklist_expires_1718`** — 定期清理过期黑名单记录时使用 `DELETE WHERE expires_at < now()`，该索引加速过期记录的定位；认证中间件也通过 `WHERE jti = ? AND expires_at > now()` 查询令牌有效性。

### 4.3 其他性能策略

#### 视图物化建议

系统中定义的 10 个视图（`view_room_details_1718`、`view_hotel_summary_1718` 等）均为普通 SQL 视图，每次查询时动态执行关联聚合计算。对于以下高频使用且数据变更不频繁的视图，可考虑在 PostgreSQL 中创建 MATERIALIZED VIEW（物化视图）并定期刷新，以换取查询性能的显著提升：

| 视图 | 理由 | 刷新策略 |
|------|------|----------|
| `view_hotel_summary_1718` | 酒店搜索列表页高频调用，聚合计算量大（MIN、AVG、COUNT） | 每日凌晨刷新 |
| `view_guest_booking_stats_1718` | 后台统计报表查询，数据准确度要求宽松 | 每小时刷新 |
| `view_order_full_1718` | 数据导出和全量分析场景，查询频率低但单次开销大 | 按需手动刷新 |

#### 连接池配置

后端使用 GORM 内置的连接池管理，配置参数如下：

| 参数 | 值 | 说明 |
|------|:---:|------|
| `SetMaxIdleConns` | 10 | 最大空闲连接数，保持数据库连接池的常驻连接，减少频繁建立连接的开销 |
| `SetMaxOpenConns` | 100 | 最大打开连接数，防止突发流量耗尽数据库连接资源 |
| `SetConnMaxLifetime` | 1 小时 | 连接最大存活时间，定期轮换连接以避免长连接导致的资源泄漏 |

该配置平衡了连接复用率和资源消耗，在典型的高校课程设计并发量下（数十人同时访问）绰绰有余。

#### GORM 性能优化

**PrepareStmt 预编译**：GORM 在 `PrepareStmt` 模式下会将 SQL 语句预编译为 Prepared Statement，数据库需要缓存查询计划并重复使用。开启后，相同 SQL 模板（仅参数不同）的查询跳过 SQL 解析和查询计划生成阶段，在高频查询场景下可减少 20%~30% 的执行时间。

**SkipDefaultTransaction**：GORM 默认对 `CREATE`、`UPDATE`、`DELETE` 等写操作包裹事务，但事务会引入额外的开销（BEGIN/COMMIT 往返）。对于单行 INSERT 等已经原子化的操作，`SkipDefaultTransaction = true` 可避免不必要的事务开销。只在需要跨表一致性的业务操作（如创建订单同时写入 `order_guest_1718`）时使用显式事务。

#### 软删除索引过滤

系统核心业务表（`user_1718`、`hotel_1718`、`room_1718`）采用软删除策略，通过 `status` 字段（1=活跃，0=已删除）标记逻辑删除。所有业务查询均携带 `WHERE status = 1` 条件。由于已删除记录在数据量积累后会占据相当比例（尤其酒店和房型表），建议在频繁查询的索引上添加 `WHERE status = 1` 条件创建**部分索引（Partial Index）**，使索引仅索引活跃记录，减小索引体积、提升查询效率。举例：

```sql
-- 当前索引（索引全部记录）
CREATE INDEX idx_hotel_region_1718 ON hotel_1718 (region_id);

-- 优化建议：部分索引（仅索引活跃记录）
CREATE INDEX idx_hotel_region_active_1718 ON hotel_1718 (region_id) WHERE status = 1;
```

#### UUID v7 时间有序优势

系统主键采用 UUID v7 格式，该版本将时间戳编码在高位比特中，使得 UUID 值天然按创建时间递增有序。与 UUID v4（完全随机）相比，UUID v7 有以下性能优势：

- **索引插入效率高**：B-Tree 索引在插入递增有序的主键时，新值总是追加到最右侧叶子节点，避免了随机插入导致的页分裂和索引重组开销。
- **索引碎片少**：有序插入使索引页保持紧凑，减少磁盘 I/O 和缓存未命中。
- **按时间排序无需额外索引**：主键本身就反映了创建顺序，对于按时间排序的查询（如倒序显示订单），可直接利用主键索引天然的有序性。

因此系统的核心表（`user_1718`、`hotel_1718`、`room_1718`、`order_1718`、`review_1718`）均使用 `DEFAULT uuidv7()` 作为主键生成策略，兼顾了全局唯一性和性能。

### 5.4 创建存储过程/函数

系统共定义了 2 个存储函数和 1 个存储过程、1 个辅助函数，以下逐一说明。

#### 1. `fn_query_orders_by_user_1718(p_username TEXT, p_id_card CHAR(18))` — 按用户信息查询订单

- **参数**：`p_username` — 用户名（模糊匹配，可选）；`p_id_card` — 身份证号（精确匹配，可选）
- **返回**：订单ID、入住/离店日期、状态、总价、实付价、创建时间、酒店名、房型名、数量
- **功能**：支持按用户名模糊匹配或身份证号精确匹配查询关联订单。先通过身份证号在 `order_guest_1718` 中查找关联的预订用户ID，再合并用户名匹配的用户ID，最终返回这些用户的所有订单详情。

```sql
CREATE OR REPLACE FUNCTION fn_query_orders_by_user_1718(
    p_username TEXT DEFAULT NULL,
    p_id_card CHAR(18) DEFAULT NULL
) RETURNS TABLE (
    order_id UUID, check_in_date DATE, check_out_date DATE,
    status VARCHAR(20), total_price DECIMAL(10,2), actual_price DECIMAL(10,2),
    create_at TIMESTAMPTZ, hotel_name TEXT, room_type TEXT, quantity INT
) LANGUAGE plpgsql AS $$
DECLARE v_user_ids UUID[];
BEGIN
    IF p_id_card IS NOT NULL THEN
        SELECT ARRAY_AGG(DISTINCT o.user_id) INTO v_user_ids
        FROM order_1718 o JOIN order_guest_1718 og ON og.order_id = o.id
        WHERE og.id_card = p_id_card;
        IF v_user_ids IS NULL THEN RAISE NOTICE '未找到身份证号 % 关联的订单', p_id_card; RETURN; END IF;
    END IF;
    IF p_username IS NOT NULL THEN
        SELECT ARRAY_AGG(id) INTO v_user_ids
        FROM (SELECT id FROM user_1718 WHERE username ILIKE '%' || p_username || '%'
              UNION SELECT UNNEST(v_user_ids)) t;
    END IF;
    IF v_user_ids IS NULL THEN RAISE NOTICE '未找到匹配的用户'; RETURN; END IF;
    RETURN QUERY
    SELECT o.id, o.check_in_date, o.check_out_date, o.status, o.total_price, o.actual_price,
           o.create_at, h.hotel_name, rm.type_name, o.quantity::INT
    FROM order_1718 o
        JOIN room_1718 rm ON rm.id = o.room_id
        JOIN hotel_1718 h ON h.id = rm.hotel_id
    WHERE o.user_id = ANY (v_user_ids)
    ORDER BY o.create_at DESC;
END;
$$;
```

#### 2. `fn_room_occupancy_rate_1718(p_hotel_id UUID, p_start_date DATE, p_end_date DATE)` — 客房入住率统计

- **参数**：`p_hotel_id` — 酒店UUID；`p_start_date` — 统计开始日期；`p_end_date` — 统计结束日期
- **返回**：房型ID、房型名、总数量、可预订数量、已预订房晚数、入住率百分比、单价、总营收
- **功能**：入住率 = 已预订房晚数 ÷（总客房数 × 统计天数）× 100。支持按日期范围精确计算跨天订单的房晚分摊。

```sql
CREATE OR REPLACE FUNCTION fn_room_occupancy_rate_1718(
    p_hotel_id UUID, p_start_date DATE, p_end_date DATE
) RETURNS TABLE (
    room_id UUID, room_type TEXT, total_quantity INT, available_quantity INT,
    booked_nights BIGINT, occupancy_rate_percent DECIMAL(6,2),
    price DECIMAL(10,2), total_revenue DECIMAL(10,2)
) LANGUAGE plpgsql AS $$
DECLARE v_total_days INT;
BEGIN
    IF p_start_date > p_end_date THEN RAISE EXCEPTION '开始日期不能晚于结束日期'; END IF;
    v_total_days := p_end_date - p_start_date + 1;
    RETURN QUERY
    SELECT rm.id, rm.type_name, rm.total_quantity, rm.available_quantity,
        COALESCE(SUM(o.quantity * GREATEST(0, LEAST(o.check_out_date, p_end_date) - GREATEST(o.check_in_date, p_start_date))), 0)::BIGINT,
        CASE WHEN rm.total_quantity > 0 AND v_total_days > 0
             THEN ROUND(COALESCE(SUM(o.quantity * GREATEST(0, LEAST(o.check_out_date, p_end_date) - GREATEST(o.check_in_date, p_start_date))), 0)::DECIMAL / (rm.total_quantity * v_total_days) * 100, 2)
             ELSE 0 END,
        rm.price,
        COALESCE(SUM(o.actual_price * GREATEST(0, LEAST(o.check_out_date, p_end_date) - GREATEST(o.check_in_date, p_start_date)) / NULLIF(o.check_out_date - o.check_in_date, 0)), 0)
    FROM room_1718 rm
        LEFT JOIN order_1718 o ON o.room_id = rm.id
            AND o.status IN ('booked', 'checked_in', 'completed')
            AND o.check_in_date < p_end_date AND o.check_out_date > p_start_date
    WHERE rm.hotel_id = p_hotel_id AND rm.status = 1
    GROUP BY rm.id, rm.type_name, rm.total_quantity, rm.available_quantity, rm.price
    ORDER BY rm.type_name;
END;
$$;
```

#### 3. `sp_ensure_person_1718(p_id_card CHAR(18), p_name TEXT, p_phone VARCHAR(20))` — 确保入住人存在

- **参数**：`p_id_card` — 身份证号（必填）；`p_name` — 姓名（必填）；`p_phone` — 手机号（可选）
- **功能**：存储过程，在创建订单前调用。若入住人已存在则更新其电话号码；否则插入新记录。使用 `ON CONFLICT ... DO UPDATE` 实现幂等操作，避免 `order_guest_1718` 外键约束失败。

```sql
CREATE OR REPLACE PROCEDURE sp_ensure_person_1718(
    p_id_card CHAR(18), p_name TEXT, p_phone VARCHAR(20) DEFAULT NULL
) LANGUAGE plpgsql AS $$
BEGIN
    INSERT INTO person_1718 (id_card, name, phone)
    VALUES (p_id_card, p_name, p_phone)
    ON CONFLICT (id_card) DO UPDATE
    SET phone = COALESCE(EXCLUDED.phone, person_1718.phone);
END;
$$;
```

#### 4. `fn_order_detail_1718(p_order_id UUID)` — 订单详情查询

- **参数**：`p_order_id` — 订单UUID
- **返回**：订单ID、状态、数量、入住/离店日期、天数、总价、实付价、创建时间、预订用户名/真实姓名/电话、酒店名、房型名、房价、入住人数、入住人姓名列表
- **功能**：查询指定订单的完整详情，关联用户信息、酒店信息、房型信息，入住人姓名通过 `STRING_AGG` 聚合。

```sql
CREATE OR REPLACE FUNCTION fn_order_detail_1718(p_order_id UUID)
RETURNS TABLE (
    order_id UUID, status VARCHAR(20), quantity INT,
    check_in_date DATE, check_out_date DATE, nights INT,
    total_price DECIMAL(10,2), actual_price DECIMAL(10,2), create_at TIMESTAMPTZ,
    order_user TEXT, order_user_name TEXT, order_user_phone VARCHAR(20),
    hotel_name TEXT, room_type TEXT, room_price DECIMAL(10,2),
    guest_count BIGINT, guest_names TEXT
) LANGUAGE plpgsql STABLE AS $$
BEGIN
    RETURN QUERY
    SELECT o.id, o.status::VARCHAR(20), o.quantity, o.check_in_date, o.check_out_date,
           (o.check_out_date - o.check_in_date)::INT, o.total_price, o.actual_price, o.create_at,
           u.username, u.real_name, u.phone, h.hotel_name, rm.type_name, rm.price,
           COALESCE(g.guest_count, 0), g.guest_names
    FROM order_1718 o
        JOIN user_1718 u ON u.id = o.user_id
        JOIN room_1718 rm ON rm.id = o.room_id
        JOIN hotel_1718 h ON h.id = rm.hotel_id
        LEFT JOIN (SELECT og.order_id, COUNT(*) AS guest_count,
                          STRING_AGG(p.name, ', ' ORDER BY p.id_card) AS guest_names
                   FROM order_guest_1718 og JOIN person_1718 p ON p.id_card = og.id_card
                   WHERE og.order_id = p_order_id GROUP BY og.order_id) g ON g.order_id = o.id
    WHERE o.id = p_order_id;
END;
$$;
```

### 5.5 创建触发器

系统共定义了 9 个触发器，分为 4 个业务触发器（含 1 个校验触发器、1 个库存触发器、1 个积分触发器、1 个折扣触发器）和 5 个维护触发器。

#### 业务触发器

##### 触发器 1：`trg_review_user_match_1718` — 评价用户一致性校验

- **触发时机**：`BEFORE INSERT OR UPDATE OF user_id, order_id ON review_1718`
- **触发方式**：FOR EACH ROW
- **调用函数**：`fn_validate_review_user_1718()`
- **功能**：校验评价的 `user_id` 必须与关联订单的 `user_id` 一致，防止用户为他人的订单撰写评价。

```sql
CREATE OR REPLACE FUNCTION fn_validate_review_user_1718()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF NEW.user_id != (SELECT user_id FROM order_1718 WHERE id = NEW.order_id) THEN
        RAISE EXCEPTION '评价用户与订单预订人不一致';
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_review_user_match_1718
    BEFORE INSERT OR UPDATE OF user_id, order_id
    ON review_1718 FOR EACH ROW
    EXECUTE FUNCTION fn_validate_review_user_1718();
```

##### 触发器 2：`trg_order_room_quantity_1718` — 客房库存自动管理

- **触发时机**：`AFTER INSERT OR UPDATE OF status ON order_1718`
- **触发方式**：FOR EACH ROW
- **调用函数**：`fn_update_room_quantity_1718()`
- **功能**：订单状态变为 `booked` 或 `checked_in` 时自动扣减对应房型的可预订数量（含库存不足校验）；订单取消时自动恢复库存。

```sql
CREATE OR REPLACE FUNCTION fn_update_room_quantity_1718()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE v_available INT;
BEGIN
    IF NEW.status IN ('booked', 'checked_in')
        AND (TG_OP = 'INSERT' OR OLD.status = 'pending')
    THEN
        SELECT available_quantity INTO v_available FROM room_1718 WHERE id = NEW.room_id;
        IF v_available < NEW.quantity THEN
            RAISE EXCEPTION '客房库存不足：可用 % 间，请求 % 间', v_available, NEW.quantity;
        END IF;
        UPDATE room_1718 SET available_quantity = available_quantity - NEW.quantity, update_at = now()
        WHERE id = NEW.room_id;
    ELSIF NEW.status = 'cancelled' AND TG_OP = 'UPDATE' AND OLD.status IN ('booked', 'checked_in')
    THEN
        UPDATE room_1718 SET available_quantity = available_quantity + OLD.quantity, update_at = now()
        WHERE id = OLD.room_id;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_order_room_quantity_1718
    AFTER INSERT OR UPDATE OF status ON order_1718 FOR EACH ROW
    EXECUTE FUNCTION fn_update_room_quantity_1718();
```

##### 触发器 3：`trg_order_user_points_1718` — 用户积分与VIP等级自动更新

- **触发时机**：`AFTER INSERT OR UPDATE OF status ON order_1718`
- **触发方式**：FOR EACH ROW
- **调用函数**：`fn_update_user_points_1718()`
- **功能**：订单变为 `completed` 时将实付金额取整累加到用户积分；从 `completed` 变为其他状态时扣减对应积分；每次积分变更后自动根据当前积分更新VIP等级。

```sql
CREATE OR REPLACE FUNCTION fn_update_user_points_1718()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF NEW.status = 'completed' AND (OLD.status IS NULL OR OLD.status != 'completed') THEN
        UPDATE user_1718 SET points = points + FLOOR(NEW.actual_price)::INT, update_at = now()
        WHERE id = NEW.user_id;
    ELSIF TG_OP = 'UPDATE' AND OLD.status = 'completed' AND NEW.status != 'completed' THEN
        UPDATE user_1718 SET points = GREATEST(0, points - FLOOR(OLD.actual_price)::INT), update_at = now()
        WHERE id = OLD.user_id;
    END IF;
    UPDATE user_1718 SET vip_level = (SELECT COALESCE(MAX(level), 0) FROM vip_level_1718 WHERE min_points <= user_1718.points),
        update_at = now()
    WHERE id = COALESCE(NEW.user_id, OLD.user_id);
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_order_user_points_1718
    AFTER INSERT OR UPDATE OF status ON order_1718 FOR EACH ROW
    EXECUTE FUNCTION fn_update_user_points_1718();
```

##### 触发器 4：`trg_order_vip_discount_1718` — 下单自动应用VIP折扣

- **触发时机**：`BEFORE INSERT ON order_1718`
- **触发方式**：FOR EACH ROW
- **调用函数**：`fn_apply_vip_discount_1718()`
- **功能**：新订单插入前根据下单用户的VIP等级查询对应折扣率，自动计算 `actual_price = total_price × discount_rate`，保证折扣应用的原子性。

```sql
CREATE OR REPLACE FUNCTION fn_apply_vip_discount_1718()
RETURNS TRIGGER AS $$
DECLARE v_discount_rate DECIMAL(3,2);
BEGIN
    SELECT COALESCE(vl.discount_rate, 1.0) INTO v_discount_rate
    FROM user_1718 u JOIN vip_level_1718 vl ON vl.level = u.vip_level
    WHERE u.id = NEW.user_id;
    NEW.actual_price := NEW.total_price * v_discount_rate;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_vip_discount_1718
    BEFORE INSERT ON order_1718 FOR EACH ROW
    EXECUTE FUNCTION fn_apply_vip_discount_1718();
```

#### 维护触发器（update_at 自动刷新）

以下 5 个触发器均调用同一函数 `fn_set_update_at_1718()`，在对对应表执行 `UPDATE` 操作前自动将 `update_at` 字段刷新为当前时间戳：

```sql
CREATE OR REPLACE FUNCTION fn_set_update_at_1718()
RETURNS TRIGGER LANGUAGE plpgsql AS $$

    RETURN NEW;

$$;

**Repo 层**（`hotel_repo.go`）负责 GORM 查询构造与分页：

```go
func (r *HotelRepo) FindAll(ctx context.Context, offset, limit int,
    regionID *int, starLevel *int16, keyword string,
    minPrice, maxPrice *float64, checkInDate, checkOutDate *time.Time,
) ([]model.Hotel, int64, error) {
    var results []model.Hotel
    var total int64
    query := r.db.WithContext(ctx).Model(&model.Hotel{}).Where("status != 0")
    if regionID != nil && *regionID > 0 {
        query = query.Where("region_id = ?", *regionID)
    }
    if starLevel != nil && *starLevel > 0 {
        query = query.Where("star_level = ?", *starLevel)
    }
    if keyword != "" {
        query = query.Where("hotel_name ILIKE ?", "%"+keyword+"%")
    }
    if minPrice != nil {
        query = query.Where("EXISTS (SELECT 1 FROM room_1718 r WHERE r.hotel_id = hotel_1718.id AND r.status = 1 AND r.price >= ?)", *minPrice)
    }
    if checkInDate != nil && checkOutDate != nil {
        query = query.Where("EXISTS (SELECT 1 FROM room_1718 r WHERE r.hotel_id = hotel_1718.id AND r.status = 1 AND r.total_quantity > COALESCE((SELECT SUM(o.quantity) FROM order_1718 o WHERE o.room_id = r.id AND o.status IN ('pending','booked','checked_in','completed') AND o.check_in_date < ? AND o.check_out_date > ?), 0))", *checkOutDate, *checkInDate)
    }
    query.Count(&total)
    if offset >= 0 && limit > 0 {
        query = query.Offset(offset).Limit(limit)
    }
    err := query.Preload("Images").Preload("Region").Find(&results).Error
    return results, total, err
}
```

> 返回统一使用 `model.SendSuccess` / `model.SendError`，分页信息由 `model.WithPagination(total, page, pageSize)` 封装。Create、Update、Delete 模式类似：Bind → 校验 → 调用 repo → 统一响应。

#### 6.2.2 前端API调用模式

前端通过 Axios 实例（`client.ts`）封装对后端 `/api/v1` 的 HTTP 请求，核心是两层拦截器：

**请求拦截器** — 自动注入 JWT Token：

```ts
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("accessToken");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

**响应拦截器** — 401 自动静默刷新 Token（避免并发重复刷新）：

```ts
let isRefreshing = false;
let failedQueue: Array<{ resolve: (token: string) => void; reject: (err: unknown) => void }> = [];

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    const isAuthEndpoint = /\/auth\/(login|register)$/.test(originalRequest.url || "");
    if (error.response?.status === 401 && !originalRequest._retry && !isAuthEndpoint) {
      if (isRefreshing) {
        // 队列等待，复用第一个 refresh 的结果
        return new Promise((resolve, reject) => {
          failedQueue.push({
            resolve: (token: string) => {
              originalRequest.headers.Authorization = `Bearer ${token}`;
              resolve(api(originalRequest));
            },
            reject,
          });
        });
      }
      originalRequest._retry = true;
      isRefreshing = true;
      const refreshToken = localStorage.getItem("refreshToken");
      if (!refreshToken) { /* 清除 token 跳转登录 */ }
      try {
        const { data } = await axios.post<ApiResponse<{ accessToken: string; refreshToken: string }>>(
          "/api/v1/auth/refresh", { refreshToken }
        );
        if (data.success && data.data) {
          localStorage.setItem("accessToken", data.data.accessToken);
          localStorage.setItem("refreshToken", data.data.refreshToken);
          processQueue(null, data.data.accessToken);
          originalRequest.headers.Authorization = `Bearer ${data.data.accessToken}`;
          return api(originalRequest);
        }
      } catch (refreshError) {
        processQueue(refreshError, null);
        localStorage.removeItem("accessToken");
        localStorage.removeItem("refreshToken");
        window.location.href = "/login";
        return Promise.reject(refreshError);
      } finally { isRefreshing = false; }
    }
    return Promise.reject(error);
  },
);
```

> 前端组件通过 Pinia store 管理认证态与业务数据。路由守卫在 `router/index.ts` 的 `beforeEach` 中检查 `requiresAuth` 与 `requiresAdmin` 元信息，未登录自动重定向至 `/login?redirect=...`。

#### 6.2.3 调用存储过程示例

系统中使用了两种数据库函数/过程模式：

**（a）过程调用：`sp_ensure_person_1718`** — 确保入住人存在于人员表

PostgreSQL 定义：

```sql
CREATE OR REPLACE PROCEDURE sp_ensure_person_1718(
    p_id_card CHAR(18), p_name TEXT, p_phone VARCHAR(20) DEFAULT NULL
) LANGUAGE plpgsql AS $$
BEGIN
    INSERT INTO person_1718 (id_card, name, phone)
    VALUES (p_id_card, p_name, p_phone)
    ON CONFLICT (id_card) DO UPDATE
    SET phone = COALESCE(EXCLUDED.phone, person_1718.phone);
END;
$$;
```

Go repo 层通过 `tx.Exec("CALL sp_ensure_person_1718(?, '', NULL)", g.IDCard)` 在事务中调用，保证创建订单前入住人记录已存在，避免外键约束失败：

```go
func (r *OrderRepo) Create(ctx context.Context, order *model.Order) error {
    return r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        for _, g := range order.Guests {
            if err := tx.Exec("CALL sp_ensure_person_1718(?, '', NULL)", g.IDCard).Error; err != nil {
                return err
            }
        }
        if err := tx.Omit("Guests").Create(order).Error; err != nil {
            return err
        }
        if len(order.Guests) > 0 {
            for i := range order.Guests {
                order.Guests[i].OrderID = order.ID
            }
            if err := tx.Omit("Order", "Person").CreateInBatches(order.Guests, len(order.Guests)).Error; err != nil {
                return err
            }
        }
        return nil
    })
}
```

**（b）函数调用：`fn_order_detail_1718`** — 查询指定订单的详细信息（走主键索引，比视图全表扫描高效）

Go 层调用：

```go
func (r *OrderDetailRepo) FindDetailByOrderID(ctx context.Context, orderID uuid.UUID) (*view.OrderDetail, error) {
    var result view.OrderDetail
    err := r.db.WithContext(ctx).Raw(
        "SELECT * FROM fn_order_detail_1718(?)", orderID,
    ).Scan(&result).Error
    if err != nil { return nil, err }
    if result.OrderID == uuid.Nil { return nil, gorm.ErrRecordNotFound }
    return &result, nil
}
```

> 该函数 JOIN 了 `order_1718`、`user_1718`、`room_1718`、`hotel_1718`、`order_guest_1718`、`person_1718` 六张表，返回下单人信息、酒店名称、房型、入住人名单等完整详情，通过 `STRING_AGG` 聚合入住人姓名。

#### 6.2.4 触发器触发示例

订单创建过程中共有 **3 个触发器** 自动响应，按触发时序如下：

```
POST /api/v1/orders
    └── orderH.Create
        └── orderSvc.Create（校验 user_id / room_id / quantity / 日期）
            └── orderRepo.Create（GORM 事务内）
                ├── 1. CALL sp_ensure_person_1718（确保入住人存在）
                ├── 2. GORM INSERT INTO order_1718
                │        ↓
                │      [BEFORE INSERT] trg_order_vip_discount_1718
                │        └─ fn_apply_vip_discount_1718():
                │             SELECT discount_rate FROM vip_level_1718
                │             WHERE level = (SELECT vip_level FROM user_1718 WHERE id = NEW.user_id)
                │             → NEW.actual_price := NEW.total_price * discount_rate
                │
                │        ↓
                │      [AFTER INSERT] trg_order_room_quantity_1718
                │        └─ fn_update_room_quantity_1718():
                │             UPDATE room_1718
                │             SET available_quantity = available_quantity - NEW.quantity
                │             WHERE id = NEW.room_id
                │
                └── 3. INSERT INTO order_guest_1718（入住人关联）
```

**`trg_order_vip_discount_1718`** — BEFORE INSERT，原子计算实际价格：

```sql
CREATE OR REPLACE FUNCTION fn_apply_vip_discount_1718()
RETURNS TRIGGER AS $$
DECLARE v_discount_rate DECIMAL(3,2);
BEGIN
    SELECT COALESCE(vl.discount_rate, 1.0)
    INTO v_discount_rate
    FROM user_1718 u
    JOIN vip_level_1718 vl ON vl.level = u.vip_level
    WHERE u.id = NEW.user_id;
    NEW.actual_price := NEW.total_price * v_discount_rate;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_vip_discount_1718
    BEFORE INSERT ON order_1718
    FOR EACH ROW EXECUTE FUNCTION fn_apply_vip_discount_1718();
```

**`trg_order_room_quantity_1718`** — AFTER INSERT/UPDATE，维护库存一致性：

```sql
CREATE TRIGGER trg_order_room_quantity_1718
    AFTER INSERT OR UPDATE OF status
    ON order_1718
    FOR EACH ROW EXECUTE FUNCTION fn_update_room_quantity_1718();
```

> 该触发器对应的函数 `fn_update_room_quantity_1718` 在 INSERT 时扣减 `available_quantity`，在订单取消（`booked`/`checked_in` → `cancelled`）时恢复库存。此外 `trg_order_user_points_1718` 在订单变为 `completed` 时增加用户积分并自动更新 VIP 等级。

#### 6.2.5 完整预订流程

```
用户搜索酒店 ──→ GET /api/v1/hotels?keyword=&regionID=&starLevel=&minPrice=&maxPrice=&checkInDate=&checkOutDate=&page=1&pageSize=10
    └── HotelHandler.List → HotelRepo.FindAll（ILIKE 模糊搜索 + 区域/星级筛选 + 价格子查询 + 日期余量子查询）
    └── 返回分页酒店列表（含图片、区域信息）

用户查看详情 ──→ GET /api/v1/hotels/:id
    └── HotelHandler.GetByID → HotelRepo.FindByID（Preload Rooms, Images, Region）
    └── 返回酒店详情、全部房型列表、设施信息

用户选择房型 ──→ GET /api/v1/rooms/:id
    └── HotelHandler.GetRoomByID → RoomRepo.FindByID（Preload Images, Facilities）
    └── 返回房型详情（价格、余量、设施、图片）

用户填写预订 ──→ POST /api/v1/orders
    Body: { room_id, user_id, quantity, check_in_date, check_out_date, total_price, guests: [...] }
    ├── OrderHandler.Create
    │   └── OrderService.Create ──→ 字段校验
    │       └── OrderRepo.Create（事务）
    │           ├── CALL sp_ensure_person_1718(guest.id_card, guest.name, guest.phone)
    │           │   └── ① 确保入住人存在（INSERT ON CONFLICT DO UPDATE）
    │           ├── GORM INSERT INTO order_1718
    │           │   ├── ② BEFORE INSERT：trg_order_vip_discount_1718
    │           │   │   └── actual_price = total_price × VIP 折扣率
    │           │   └── ③ AFTER INSERT：trg_order_room_quantity_1718
    │           │       └── available_quantity -= quantity
    │           └── INSERT INTO order_guest_1718（批量）
    │               └── ④ 关联入住人与订单
    │
    └── 返回 201 Created
        Response: { success: true, data: { id, status: "pending", actual_price, ... } }
```

> 整个流程中，数据库层通过 BEFORE INSERT 触发器（折扣计算）和 AFTER INSERT 触发器（库存扣减）保证关键业务规则的原子性，无需应用层加锁。存储过程 `sp_ensure_person_1718` 使用 UPSERT 语义确保入住人存在。订单状态机的合法流转由 `order_service.go` 的 `allowedTransitions` 函数严格校验：`pending → booked → checked_in → completed`（正向），`pending/booked → cancelled`（取消）。
