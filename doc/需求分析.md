# 选题四 酒店预订管理系统——需求分析

## 一、系统概述

### 1.1 项目背景

随着互联网和旅游业的快速发展，传统的酒店预订方式（如电话预订、到店预订）已无法满足现代消费者的需求。传统的线下预订模式存在诸多不便：顾客无法全面了解异地酒店情况，选择范围受限；酒店方难以掌握顾客行程变化，需频繁长途沟通；双方都面临恶意预订等风险。

因此，开发一个第三方酒店预订管理系统具有重要意义。该系统模拟携程网等OTA（在线旅游代理）平台的运营模式，为顾客提供便捷的在线酒店预订服务，同时为酒店方提供高效的订单管理和客房管理工具。

### 1.2 系统目标

本系统旨在实现以下目标：

1. 为顾客提供酒店信息查询、在线预订、订单管理、入住评价等一站式服务；
2. 为酒店管理者提供客房信息维护、订单管理等后台管理功能；
3. 通过会员积分和VIP等级制度提升用户粘性和忠诚度；
4. 通过数据统计与分析，为酒店营销策略制定提供决策支持。

### 1.3 系统范围

本系统是一个第三方酒店预订管理平台，涵盖酒店信息展示、客房预订、订单管理、用户管理、评价管理、数据统计等核心功能模块。系统采用B/S架构，用户通过浏览器访问系统进行各项操作。**客户入住酒店后的具体入住管理（如办理入住、退房等）不在本系统的功能范围内**。

## 二、用户角色分析

本系统涉及以下四类用户角色：

| 角色             | 描述                   | 主要职责                                 |
| ---------------- | ---------------------- | ---------------------------------------- |
| **普通顾客**     | 在系统注册的普通用户   | 查询酒店、预订客房、管理订单、发表评价   |
| **VIP顾客**      | 积分达到一定等级的用户 | 享有预订优惠等特权                       |
| **酒店管理人员** | 各酒店在系统上的管理员 | 维护酒店基本信息、管理客房信息、查看订单 |
| **系统管理员**   | 平台方的超级管理员     | 用户管理、酒店管理、系统配置             |

## 三、功能需求

### 3.1 用户管理模块

**3.1.1 用户注册**

- 用户必须注册并登录后方可使用系统。
- 注册信息包括：用户名、密码、手机号、邮箱、姓名、身份证号等。
- 系统对用户密码进行加密存储，确保安全性。

**3.1.2 用户登录**
****

- 用户通过用户名/手机号和密码登录系统。
- 未登录用户无法进行预订、评价等操作。

**3.1.3 用户信息维护**

- 用户可以查询和修改个人信息（如手机号、邮箱等）。
- 用户可以查看自己的积分和VIP等级。

**3.1.4 积分与VIP制度**

- 用户每次成功预订并完成入住后可获得积分。
- 积分累计达到一定标准可升级为VIP用户。
- VIP用户预订酒店可享受折扣优惠（如95折、9折等），优惠策略由系统设定。

### 3.2 酒店与客房管理模块

**3.2.1 酒店信息管理**

酒店基本信息包括：

- 酒店ID（主键）
- 酒店名称
- 酒店地址（城市、区域、详细地址）
- 酒店星级
- 酒店简介
- 联系电话
- 酒店图片

酒店管理人员可以维护本酒店的基本信息。

**3.2.2 客房信息管理**

客房信息包括：

- 客房ID（主键）
- 所属酒店ID（外键）
- 房型名称（如标准间、大床房、豪华套房等）
- 房间数量（该房型的总量）
- 可预订数量（总数量减去已预订数量）
- 价格（平日价、周末价等）
- 房间设施（如WiFi、空调、电视等）
- 房间图片

酒店管理人员可以增删改客房信息。

### 3.3 酒店查询与展示模块

**3.3.1 酒店搜索**

用户可以根据以下条件搜索酒店：

- 城市名
- 区域名（如商圈、行政区）
- 酒店名称（模糊查询）
- 价格范围
- 酒店星级
- 入住日期和离店日期

**3.3.2 酒店列表展示**

- 系统根据用户搜索条件返回符合条件的酒店清单。
- 列表显示：酒店名称、地址、评分、起价等信息。

**3.3.3 酒店详情展示**

- 用户点击某酒店后可查看详细信息：
  - 酒店基本信息（名称、地址、星级、简介等）
  - 所有房型及价格
  - 各房型的可预订状态
  - 酒店图片
  - 用户评价

### 3.4 客房预订模块

**3.4.1 预订流程**

1. 用户选择酒店，进入酒店详情页；
2. 选择房型、入住日期、离店日期、房间数量；
3. 填写入住人信息（姓名、身份证号等），一间客房可入住多人；
4. 确认订单信息并提交预订；
5. 系统生成订单并更新可预订客房数量。

**3.4.2 预订规则**

- 用户可为自己预订，也可为他人预订（需填写入住人实名信息）。
- 一次可预订多间客房。
- 一间客房可同时入住多人（如标准间可入住2人）。

**3.4.3 客房数量自动更新**

- 当有客房被预订时，该酒店同类型可预订客房数量自动减少。

### 3.5 订单管理模块

**3.5.1 订单信息**

订单包含以下信息：

- 订单号（系统自动生成，唯一）
- 用户ID（预订人）
- 酒店ID
- 房型ID
- 房间数量
- 入住日期
- 离店日期
- 入住人信息（姓名、身份证号等）
- 订单总价
- 折扣金额（VIP用户享受）
- 实付金额
- 订单状态（待支付、已预订、已入住、已完成、已取消等）
- 下单时间

**3.5.2 订单查询**

用户可以按以下方式查询订单：

- 全部订单
- 已预订还未入住的订单（待入住）
- 已入住的订单
- 已完成的订单（已离店）
- 已取消的订单

酒店管理人员可以查询本酒店的所有订单。

**3.5.3 订单取消**

- 对于已预订但尚未入住的订单，用户可以申请取消。
- 取消后，对应客房的可预订数量自动恢复。

### 3.6 评价模块

- 用户可以对已入住的酒店进行评价和打分。
- 评价内容包括：评分（如1-5分）、评论文本、评价时间。
- 评价信息在酒店详情页中展示，供其他用户参考。

### 3.7 统计汇总模块

**3.7.1 按酒店维度的统计**

- 按城市、区域、酒店等维度统计一段时间内客房的预订情况。
- 统计指标包括：预订数量、预订金额、入住率等。
- 为酒店制定后续营销策略提供数据支持。

**3.7.2 按客户维度的统计**

- 按入住人信息（年龄、性别、职业、受教育程度、收入状况等）统计客房预订情况。
- 分析不同客户群体的预订偏好。
- 为客户推荐合适的酒店和房型，为营销策略制定提供指导。

### 3.8 后台管理模块

**3.8.1 酒店管理（系统管理员）**

- 添加/删除酒店。
- 为酒店分配酒店管理人员账号。

**3.8.2 客房管理（酒店管理人员）**

- 增删改本酒店的客房信息。
- 实时更新客房状态和可预订数量。

**3.8.3 用户管理（系统管理员）**

- 查询所有用户信息。
- 禁用/启用用户账号。

## 四、非功能需求

### 4.1 易用性要求

- 交互界面简洁友好，操作流程清晰。
- 多种输入形式：文本框输入、下拉选择、单选、复选等。
- 酒店和客房信息应有图形化显示（图片展示）。

### 4.2 容错性要求

- 日期输入需进行有效性校验（如不能输入2021-5-35等非法日期）。
- 对输入的不合理数据系统应有相应提示，而非程序中断运行。
- 价格输入不能为负数。

### 4.3 安全性要求

- 用户密码需加密存储。
- 只有合法登录的用户才能进行预订、评价等操作。
- 不同角色用户具有不同的操作权限。

### 4.4 性能要求

- 系统应能支持多用户并发访问。
- 酒店查询和预订操作响应时间应在可接受范围内。

## 五、数据需求概述

本系统涉及的主要数据实体包括：

1. **用户（User）** ：用户ID、用户名、密码、手机号、邮箱、姓名、身份证号、积分、VIP等级、注册时间等。
2. **酒店（Hotel）** ：酒店ID、名称、城市、区域、详细地址、星级、简介、联系电话、图片等。
3. **客房（Room）** ：客房ID、酒店ID、房型名称、总数量、可预订数量、价格、设施、图片等。
4. **订单（Order）** ：订单号、用户ID、酒店ID、房型ID、房间数量、入住日期、离店日期、入住人信息、总价、折扣、实付金额、订单状态、下单时间等。
5. **评价（Review）** ：评价ID、订单ID、用户ID、酒店ID、评分、评论文本、评价时间等。
6. **入住人（Guest）** ：入住人ID、姓名、身份证号、性别、年龄、职业、受教育程度、收入状况等（与订单关联）。

各实体之间通过主键和外键建立关联关系，确保数据的完整性和一致性。
